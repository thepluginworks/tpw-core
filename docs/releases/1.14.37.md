This release standardises how TPW Core assigns society IDs for single-site installs. New member and household records now use one clear site-level value instead of relying on mixed defaults or inferred historical data, which makes member creation and signup completion more predictable for site owners and administrators.

### Main changes

- Added a Core-owned site setting, `tpw_site_society_id`, as the canonical source of truth for the site's entity society ID.
- Defaulted that site-level value to `1` for current single-site installs during activation and normal plugin bootstrap when the option is missing.
- Updated signup finalization so newly completed Join requests create members with the canonical site society value when no explicit society context is already present.
- Updated manual member creation and create-from-user flows so new member records no longer drift between hardcoded values, empty fallbacks, or inferred member data.
- Updated household creation and related household ownership checks so new real household rows also resolve to the canonical positive site society value instead of defaulting to `0`.
- Kept scope intentionally tight: this release does not redesign multi-society architecture or rewrite existing legacy rows that already have `society_id = 0`.

If your site already contains older records with `society_id = 0`, those rows can still be backfilled in a separate migration. This release focuses on making all new Core-created member and household entities consistent going forward.
